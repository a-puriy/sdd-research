# エージェント活用によるRPIフロー導入検討資料

**作成日**: 2026-03-22
**対象**: コンサルティング企業 シニアマネージャー向け
**目的**: 30名体制スクラムチームにおけるエージェント活用RPIフロー導入の妥当性評価

---

## エグゼクティブサマリー

本資料は、15名オンサイト（コードレビュー担当）＋15名オフショア（アプリ開発担当）で構成される30名体制のスクラムチームにおいて、AIエージェントを活用したRPI（Research-Plan-Implement）フローの導入妥当性を検討するものである。

SpecKitの試験運用で顕在化した「膨大なドキュメントレビュー負荷」を解決するため、汎用フレームワークの直接適用ではなく、PJ固有のワークフロー最適化を目指す。本資料では、主要SDDフレームワーク（SpecKit/OpenSpec/Kiro）の設計思想を分析し、RPIフロー設計において重要となる観点、導入評価の方法論、および長期的な管理指針を示す。

---

## 1. 現状の課題構造

### 1.1 SpecKit試験運用で判明した問題

SpecKitは「Constitution → Specify → Plan → Checklist → Tasks → Implement → Analyze」の多段階ワークフローを持ち、各段階で構造化されたMarkdownアーティファクトを生成する。この設計は仕様の網羅性を担保する一方、以下の問題を引き起こした。

**レビュー負荷の二重化**: Martin Fowlerサイト上で[Birgitta Böckeler（Thoughtworks Distinguished Engineer）が指摘する通り](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html)、SDDツールは「冗長で繰り返しの多いMarkdown成果物」を生成し、コードレビューに加えて仕様レビューが発生する。SpecKitの場合、小規模なバグ修正であっても4ユーザーストーリー・16受入基準が生成されるケースがあり、レビュー対象の量がタスクの規模に比例しない。

**Approval Fatigue**: 全フェーズにゲートを設けるワークフローでは、レビュアーが形骸的な承認を行う「承認疲れ」が発生する。特にオンサイト15名が仕様レビューとコードレビューの双方を担当する現体制では、1スプリントあたりのレビュー量がボトルネックとなる。

**Context Rotの蓄積**: 仕様書が長大になるほど、エージェント自身がコンテキストウィンドウの制約により仕様を無視・誤解する確率が上昇する。[Böckelerの報告](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html)では、拡張されたコンテキストウィンドウでもエージェントが仕様を遵守しない事象が複数確認されている。

### 1.2 問題の本質：プロセスの粒度とタスク特性の不一致

これらの問題の根本原因は、汎用フレームワークが「タスクの規模・性質に関わらず均一なプロセス」を適用する点にある。運用中プロダクトの機能追加・リファクタリングが主体のPJでは、変更の大半は既存パターンの延長であり、フルスペック化の必要がないものが多い。一方で、アーキテクチャに影響する変更には厳密な仕様化が不可欠である。この粒度の不一致を解消することが、RPIフロー設計の最重要課題となる。

## 2. 主要フレームワークの設計思想比較と抽出すべき観点

### 2.1 3フレームワークの設計思想の対比

以下に、SpecKit・OpenSpec・Kiroの設計思想を「RPIフローに活用できる要素」の観点で整理する。

| 観点 | SpecKit | OpenSpec | Kiro |
|------|---------|---------|------|
| **プロセスの剛性** | 高（Constitution→多段階ゲート） | 低（Explore→Propose→Apply、流動的） | 中（3段階ゲート＋`-y`による省略可） |
| **仕様の位置づけ** | 一次成果物（Spec-anchored寄り） | 変更単位のデルタ仕様 | 一次成果物（Spec-first） |
| **レビュー負荷の設計** | 品質チェックリストで網羅性を追求 | アーティファクトは最小限 | フェーズごとに絞り込み |
| **並列性の扱い** | タスクレベルで`[P]`マーキング | 明示的な並列分析なし | タスクレベルで`(P)`マーキング |
| **コンテキスト管理** | Constitution＋テンプレート | 変更単位のスコープ限定 | Steering（プロジェクト記憶） |
| **既存コードベースとの親和性** | 中（Constitution設定が前提） | 高（デルタ仕様で軽量） | 高（Steering＋Gap分析） |

### 2.2 RPIフロー設計で重要となる5つの観点

フレームワーク分析と外部リファレンスから、RPIフロー設計において以下の5観点が重要であると判断する。

#### 観点1: タスク規模に応じたプロセスの可変性

**なぜ重要か**: [Böckelerが指摘する](https://martinfowler.com/articles/exploring-gen-ai/sdd-3-tools.html)「小さなバグに16受入基準」問題の直接的な解法。[Block社のRPIフレームワーク](https://engineering.block.xyz/blog/ai-assisted-development-at-block)も、全タスクに均一なプロセスを適用していない。

**具体的な設計方針**: 変更の影響範囲に応じて3段階のワークフローを定義する。

- **Lightweight（影響範囲：単一ファイル/関数）**: 口頭またはチャットで意図共有 → 直接実装 → コードレビュー。RPIのうちPlanのみ省略形で実施。
- **Standard（影響範囲：単一モジュール/サービス）**: Research（既存コード調査） → Plan（実装方針1ページ以内） → Implement → コードレビュー。RPIフルサイクルだが各段階の出力量を制限。
- **Full（影響範囲：複数モジュール/アーキテクチャ変更）**: Research（Gap分析含む） → Plan（設計書＋タスク分解） → フェーズゲートレビュー → Implement → コード＋設計レビュー。

**判断の具体基準**: 曖昧さを排除するため、以下の判定チェックリストを使用する。

| 判定項目 | Yes → レベルを上げる |
|---------|-------------------|
| 変更対象ファイルが3ファイル超か | Lightweight → Standard |
| 既存のpublic APIシグネチャを変更するか | Lightweight → Standard |
| DBスキーマ変更を伴うか | Standard → Full |
| 他チーム/サービスとのインターフェースに影響するか | Standard → Full |
| 新しい外部依存（ライブラリ/API）を追加するか | Standard → Full |

スプリントプランニング時にPO/SMがこのチェックリストで分類し、判断に迷う場合は上位レベルを適用する。この基準自体もレトロスペクティブで見直し対象とする。

#### 観点2: レビューのスコープ分離と機械的品質矯正

**なぜ重要か**: レビュー負荷の二重化問題に対し、「何を人間がレビューし、何を機械に任せるか」の線引きが不可欠。[逆瀬川の分析](https://nyosegawa.com/posts/coding-agent-workflow-2026/)では、Linter/Hooks/型チェッカーによる機械的品質矯正がレビュー工数を構造的に削減すると指摘されている。

**具体的な設計方針**:
- **機械的検証（自動化対象）**: 型チェック、Lint、テスト通過、コーディング規約準拠、仕様-タスク間のトレーサビリティ整合性。これらはCIパイプラインとpre-commitフックで担保し、人間のレビュー対象から除外する。
- **人間によるレビュー（焦点を絞る）**: アーキテクチャ判断の妥当性、ビジネスロジックの正確性、エッジケースの考慮、命名・設計の意図理解。
- **エージェントによるレビュー補助**: PRサマリーの自動生成、変更影響範囲の可視化、既知パターンとの差分検出。[Block社の実績](https://engineering.block.xyz/blog/ai-assisted-development-at-block)では自動PR作成が21倍に増加している。

#### 観点3: コンテキストの階層的管理とProgressive Disclosure

**なぜ重要か**: Context Rot（コンテキスト肥大化による出力品質劣化）は、仕様駆動開発の構造的弱点である。[逆瀬川の分析](https://nyosegawa.com/posts/coding-agent-workflow-2026/)では、コンテキスト利用率40-60%での予防的圧縮が推奨されている。[Anthropicのエージェントスキル設計](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills)でも、3層のProgressive Disclosureにより使用されないスキルのトークン消費を98%削減している。

**具体的な設計方針**: Kiroのsteering概念を採用しつつ、以下の3層構造で情報を管理する。

1. **常時ロード層（CLAUDE.md/AGENTS.md）**: プロジェクト全体の規約、命名規則、ディレクトリ構造パターン。100-200行以内。
2. **フェーズロード層（steering/）**: 技術スタック詳細、API規約、テスト方針。Research/Planフェーズでのみ参照。
3. **オンデマンド層（スキル/リファレンス）**: 特定ドメインの詳細知識。エージェントが必要と判断した場合のみ動的ロード。

Kiroのsteering-principlesにある「新しいコードが既存パターンに従うならSteeringの更新は不要」という原則は、運用中プロダクトの保守・機能追加において特に有効である。

#### 観点4: オンサイト-オフショア間の責務境界の明確化

**なぜ重要か**: RPIの各フェーズをどちらのチームが主導するかの設計が、ワークフロー全体の成否を左右する。[Boris Taneのアノテーションサイクル](https://boristane.com/blog/how-i-use-claude-code/)（計画→レビュー→注釈修正→再返却を1-6回繰り返す）は、「計画承認前の実装開始を構造的に防止する」仕組みとして参考になる。

**具体的な設計方針**:

| RPIフェーズ | 主導 | 補助 | 成果物 |
|------------|------|------|--------|
| **Research** | オンサイト or オフショア（ドメイン知見による） | エージェント | 調査メモ（既存コード分析、影響範囲、制約） |
| **Plan** | 起案者（R担当者と同一） → 相手側がレビュー・質問 | エージェント | 実装方針書（Standardは1ページ、Fullは設計書） |
| **Plan承認** | オンサイト（最終判断） | — | 承認記録 |
| **Implement** | オフショア | エージェント | コード＋テスト |
| **Code Review** | オンサイト | エージェント（PR要約・差分検出） | レビューコメント |

この設計のポイントは2つある。

第一に、**Research/Planの起案者を固定しない**ことである。当該領域のドメイン知識が蓄積されているメンバ（オンサイト・オフショア問わず）が起案し、もう一方がレビュー・質問を行う。現実のオフショア開発では、開発者側に特定モジュールの深い知見があるケースが多い。起案を一律にオンサイトに固定すると、オンサイトのPlan起案工数が新たなボトルネックになり、かつ品質も低下する。ただし、**Plan承認（ゲート判断）はオンサイトが担う**ことで、品質統制の一貫性を維持する。

第二に、Plan段階での**双方向コミュニケーション**を構造に組み込むことである。従来の「オフショアが実装→オンサイトがレビュー」の一方向フローでは、手戻り発生時のコスト（タイムゾーン差による遅延含む）が大きい。Plan段階でのQ&Aにより、Implement段階での手戻りを構造的に削減する。

#### 観点5: 段階的導入とフィードバックループの組み込み

**なぜ重要か**: [ThoughtWorksのTechnology Radar](https://www.thoughtworks.com/en-us/radar/techniques/spec-driven-development)は、SDDを「Assess（評価）」段階に位置づけており、全チームへの一斉展開ではなく慎重な評価を推奨している。また、ThoughtWorksが警告する「Bitter Lesson」（AIに対する手作りの詳細ルールは最終的にスケールしない）を考慮し、プロセスの過剰な精巧化を避ける必要がある。

**具体的な設計方針**:
- **Phase 0（2-4週間）**: オンサイト3-5名をAI Championsとして選出（[Block社モデル](https://engineering.block.xyz/blog/ai-assisted-development-at-block)）。既存リポジトリにCLAUDE.md/steering整備。Lightweightワークフローのみ試行。
- **Phase 1（1-2スプリント）**: パイロットチーム（Champion＋オフショア3-5名）でStandardワークフローを試行。レビュー負荷・手戻り率・開発速度を計測。
- **Phase 2（2-4スプリント）**: パイロット結果に基づきワークフローを調整。Fullワークフローの試行。全チームへの段階的展開。
- **Phase 3（継続）**: スプリントレトロスペクティブでワークフロー自体を改善対象に含める。モデル進化に合わせたプロセスの簡素化（Bitter Lesson対策）。

## 3. 導入評価の方法論

### 3.1 定量評価指標

RPIフロー導入の効果を測定するため、以下の指標を設定する。[Block社の実績](https://engineering.block.xyz/blog/ai-assisted-development-at-block)（AI著作コード量69%増、時間削減37%増、自動PR作成21倍増、95%のエンジニアが定常利用）を参考値とするが、プロダクト特性・チーム規模の差異があるため、自チームのベースライン比での改善率を重視する。

| 指標カテゴリ | 指標 | 計測方法 | 評価基準 |
|-------------|------|---------|---------|
| **レビュー負荷** | 1PRあたりレビュー所要時間 | GitHubのPRオープン→承認までの時間 | ベースライン比20%以上削減で有効 |
| **レビュー負荷** | 1スプリントあたりレビュー件数/人 | スプリント実績 | 増加しても所要時間が減少していれば正 |
| **手戻り率** | PR差し戻し率（Request Changes / 全PR） | GitHub PR統計 | ベースライン比で減少傾向 |
| **手戻り率** | Implement後の設計変更発生率 | タスク管理ツール | Plan段階の充実度の代理指標 |
| **開発速度** | ストーリーポイント消化速度 | スプリント実績 | ベースライン比で改善傾向 |
| **開発速度** | リードタイム（チケット起票→マージ） | タスク管理ツール＋GitHub | Standard/Full別に計測 |
| **品質** | 本番障害発生率（変更起因） | インシデント管理 | 増加しないことが最低条件 |
| **品質** | テストカバレッジ変化 | CI計測 | 低下しないことが最低条件 |
| **AI活用度** | co-authored-byコミット比率 | git log解析 | 継続的な増加傾向 |
| **AI活用度** | エージェント利用セッション数/人・週 | ツール利用ログ | 定着度の指標 |

#### 計測上の留意点

- **ベースライン期間**: 導入前2-4スプリント分のデータを収集する。ベースラインなしでは改善効果を主張できない。
- **ホーソン効果の考慮**: パイロットチームは注目されることで一時的に生産性が向上する傾向がある。2スプリント以上の継続計測で安定傾向を確認する。
- **外部要因の統制**: スプリント間でタスクの難易度・規模が変動するため、ストーリーポイントの絶対値ではなく「ポイントあたりの実工数」で正規化する。

### 3.2 定性評価指標

定量指標だけでは捉えられない側面を、以下の方法で定性的に評価する。

| 評価観点 | 評価方法 | 頻度 |
|---------|---------|------|
| **レビュー体験の質** | レビュアー向けアンケート（5段階）: 「仕様の理解しやすさ」「レビュー焦点の明確さ」「承認判断のしやすさ」 | スプリント毎 |
| **開発者体験** | オフショア開発者向けアンケート: 「Plan文書の明確さ」「エージェント活用の有効性」「手戻りの予測可能性」 | スプリント毎 |
| **コミュニケーション品質** | オンサイト-オフショア間のPlan段階での質問・確認のやり取り量と質 | サンプリング（月1回、5PR抽出） |
| **プロセス適合性** | 「ワークフローレベルの選択が適切だったか」のレトロスペクティブ振り返り | スプリント毎 |
| **ナレッジ蓄積** | steering/CLAUDE.mdの更新頻度と内容の充実度 | 月次 |

#### 定性評価の活用方法

定性データは「なぜ定量指標がそのように変化したか」の解釈に使用する。例えば、レビュー所要時間が減少しても、レビュアーが「形骸的な承認が増えた」と感じていれば、品質リスクが潜在している。定量・定性の両面で改善傾向が確認できた場合にのみ、展開フェーズに進む判断とする。

### 3.3 判断基準の設定

導入の継続・拡大・中止の判断基準を事前に定めておく。

- **拡大条件**: レビュー所要時間20%以上削減 AND 本番障害率が増加していない AND レビュアー体験アンケートが3.5/5.0以上。20%の閾値は、ワークフロー変更に伴う学習コスト（推定10-15%の一時的な生産性低下）を上回る改善が確認できる水準として設定。パイロット結果に応じて調整する
- **継続条件（要調整）**: いずれかの定量指標が改善傾向 AND 定性評価で深刻な懸念なし
- **中止条件**: 本番障害率が有意に増加 OR レビュアー体験が2.0/5.0以下 OR パイロットチームから継続拒否の意見が過半

## 4. Commands・Skills・Agentsの設計指針

RPIフローを支えるClaude Code（またはそれに類するコーディングエージェント）のカスタマイズ層について、PJメンバが長期的に管理可能な設計指針を示す。

### 4.1 設計思想：3つの原則

フレームワーク分析から抽出した、長期運用に耐える設計原則を以下に示す。

#### 原則1: Steeringファーストの情報設計

SpecKitのConstitution、Kiroのsteering、OpenSpecのコンテキストファイルはすべて「エージェントにプロジェクト知識を提供する」機構であるが、粒度と管理コストが異なる。

- **SpecKitのConstitution**: 不変の原則を定義する。変更にはsemantic versioningが必要で、ガバナンスが重い。運用中プロダクトの継続的な変化に対して硬直的。
- **Kiroのsteering**: パターンレベルで記述し、「新しいコードが既存パターンに従うなら更新不要」という原則で管理コストを抑制。Bootstrap（自動生成）＋Sync（差分更新）モードで保守を半自動化。
- **OpenSpecのコンテキスト**: 変更単位でスコープされ、最も軽量だが横断的な知識の蓄積に弱い。

**推奨**: Kiroのsteering方式を基盤とし、以下のファイル構成で運用する。

```
.kiro/steering/
  product.md      # プロダクトの目的・ビジネスコンテキスト（〜50行）
  tech.md         # 技術スタック・主要ライブラリ・規約（〜100行）
  structure.md    # ディレクトリ構成パターン・命名規則（〜80行）
  api-standards.md  # API設計規約（必要に応じて）
  testing.md        # テスト方針（必要に応じて）
```

管理ルール:
- 「パターン > 列挙」を徹底する。50個のファイルパスを列挙するのではなく、「feature-first organization」のようにパターンとして記述する。
- 更新トリガーは「既存パターンに収まらない新しいアーキテクチャ判断が行われた時」のみ。日常的な機能追加では更新不要。
- AI Championsがsteering保守の責務を担い、月次で陳腐化チェックを行う。

#### 原則2: コマンドの薄い抽象層

SpecKitは8コマンド、Kiroは10コマンド、OpenSpecは4コマンドを定義しているが、コマンド数の多さは「どのコマンドをいつ使うか」の学習コストを増大させる。

**推奨**: RPIフローに必要な最小コマンドセットとして以下の5コマンドを定義する。

| コマンド | 用途 | 入力 | 出力 |
|---------|------|------|------|
| `/research` | 既存コード調査・影響範囲分析 | チケット/課題の説明 | 調査メモ（制約・依存・リスク） |
| `/plan` | 実装方針の策定 | 調査メモ＋要件 | 実装方針書（レベルに応じた粒度） |
| `/implement` | 計画に基づく実装 | 承認済み方針書 | コード＋テスト |
| `/review-assist` | PRレビューの補助 | PR差分 | 変更要約・影響分析・懸念点 |
| `/status` | 進捗確認 | — | 現在のフェーズ・完了タスク・ブロッカー |

なお、Lightweightワークフローではこれらのコマンドを使用せず、開発者が直接エージェントと対話して実装する。コマンドはStandard/Fullワークフローで「プロセスの再現性を担保する」ためのものであり、全作業にコマンド利用を強制するものではない。

コマンドの内部実装は段階的に洗練するが、インターフェース（コマンド名と入出力の期待値）は安定させる。「中身は変わっても使い方は変わらない」状態を維持することで、30名のチームメンバへの再教育コストを抑制する。

#### 原則3: スキルによるドメイン知識のカプセル化

[Anthropicのエージェントスキル設計](https://www.anthropic.com/engineering/equipping-agents-for-the-real-world-with-agent-skills)が示すProgressive Disclosure（3層ネスト構造）は、大規模チームでの運用に適している。

**推奨**: 以下の粒度でスキルを設計する。

- **チーム共通スキル**: コーディング規約チェック、PRテンプレート生成、テストパターン適用。全メンバが利用。SKILL.mdに要約のみ記載し、詳細は補助ファイルに格納（Progressive Disclosure）。
- **ドメイン固有スキル**: 特定モジュールの業務ロジック、外部API連携パターン。該当領域の作業時のみロード。
- **ワークフロースキル**: RPIの各フェーズの実行手順。コマンドの内部実装として機能。

管理ルール:
- スキルの作成・更新はAI Championsが担当する。一般メンバはスキルの利用者であり、作成者ではない。
- スキルの有効性は「利用頻度」と「利用後の手戻り率」で評価する。利用されないスキルは削除する。
- 1スキルあたりのSKILL.mdは50行以内。肥大化したスキルは分割する。

### 4.2 長期管理のための運用ガイドライン

#### 誰が何を管理するか

| 管理対象 | 責任者 | 更新頻度 | レビュー |
|---------|--------|---------|---------|
| CLAUDE.md | AI Champions（オンサイト） | 月次またはアーキテクチャ変更時 | Championsのピアレビュー |
| steering/ | AI Champions（オンサイト） | 月次の陳腐化チェック | Championsのピアレビュー |
| コマンド定義 | AI Champions（オンサイト） | 四半期またはモデル更新時 | SM＋Champions |
| スキル | AI Champions（オンサイト）＋ドメインエキスパート | 必要に応じて | Champions |
| ワークフローレベル基準 | SM＋PO | スプリント開始時（適用） | レトロスペクティブ |

#### 陳腐化防止の仕組み

Kiroのsteering-principlesが示す「パターン > 列挙」の原則に加え、以下を実施する。

1. **四半期レビュー**: AI Championsがsteering/コマンド/スキルの全量を確認し、実態との乖離がないか検証する。
2. **利用ログの分析**: コマンド・スキルの利用頻度を追跡し、利用されていないものを棚卸しする。
3. **モデル更新対応**: AIモデルのメジャーアップデート時に、既存のプロンプト設計が有効かどうかを検証する。[ThoughtWorksが警告する「Bitter Lesson」](https://www.thoughtworks.com/en-us/radar/techniques/spec-driven-development)（手作りの詳細ルールはモデル進化でスケールしなくなる）を念頭に、プロンプトの簡素化が可能であれば積極的に行う。

#### Bitter Lesson対策：過剰な精巧化の回避

SDDフレームワークの最大のリスクは、プロセスの精巧化がモデル能力の進化速度を下回り、「精巧なプロセスが足かせになる」状態である。これを防ぐため、以下の原則を設定する。

- **プロンプトの行数制限**: 1コマンドあたり100行以内。SpecKitの一部コマンドは300行超のプロンプトを持つが、これはモデル更新時のメンテナンスコストが高い。
- **ルールの最小化**: 「やるべきこと」よりも「やってはいけないこと」（ネガティブルール）を優先する。ポジティブルールの多くはモデルの能力向上で不要になるが、ネガティブルール（セキュリティ制約、ビジネスルール）は残存する。
- **定期的な簡素化レビュー**: 四半期ごとに「このルール/プロンプトはまだ必要か？モデルが自律的に判断できるようになっていないか？」を検証する。

## 5. リスクと対策

### 5.1 導入リスクの整理

| リスク | 発生確率 | 影響度 | 対策 |
|-------|---------|--------|------|
| **レビュー負荷が減少せず増加する** | 中 | 高 | ワークフローレベル分類の厳格化。Lightweightの適用範囲を広げる。機械的検証の自動化範囲を拡大 |
| **オフショアがPlanを理解せず手戻り多発** | 中 | 高 | Plan段階での双方向Q&Aセッションの義務化。FAQのスキル化 |
| **エージェントが仕様を無視する** | 高 | 中 | コンテキストの階層化で入力量を制限。タスク粒度の細分化（2-5分単位）。テストによる事後検証 |
| **AI Champions の負荷集中** | 中 | 中 | Champions を30%の時間割り当て（[Block社モデル](https://engineering.block.xyz/blog/ai-assisted-development-at-block)）。Championsの回転制（四半期ごと） |
| **モデル更新でプロンプトが壊れる** | 高 | 低-中 | Bitter Lesson対策の実施。プロンプトの簡素化。更新時の検証プロセス |
| **チームの抵抗・定着しない** | 中 | 高 | Phase 0でのChampions先行体験。ゲーミフィケーション（[Block社のRepo Quest](https://engineering.block.xyz/blog/ai-assisted-development-at-block)参考）。成功事例の共有 |

### 5.2 撤退基準の事前定義

導入が想定通りに進まない場合の撤退基準を事前に定めることで、サンクコストバイアスによる継続判断を防止する。

- **Phase 1終了時点**: パイロットチームの定量指標が全項目でベースライン以下 → RPIフローの根本的な再設計、または導入中止の判断
- **Phase 2終了時点**: 展開チームの本番障害率が有意に増加 → 展開の一時停止と原因分析
- **いつでも**: レビュアーの過半数から「以前の方が良かった」という明確なフィードバック → プロセスの簡素化または部分撤退

## 6. 推奨アクションプラン

### 即座に実施可能（Phase 0: 2-4週間）

1. **AI Champions 3-5名を選出する**。オンサイトメンバから、エージェント活用に関心が高く技術力のあるメンバを選ぶ。30%の時間をRPIフロー整備に充当する。
2. **既存リポジトリにCLAUDE.md＋steering/を整備する**。ChampionsがKiroのsteering-bootstrap相当の作業を実施。product.md・tech.md・structure.mdの3ファイルを作成する。
3. **ベースラインデータを収集する**。2-4スプリント分のレビュー所要時間・手戻り率・リードタイムを記録する。

### パイロット実施（Phase 1: 1-2スプリント）

4. **パイロットチームを編成する**。Champions 2-3名＋オフショア3-5名。
5. **5コマンドの初期版を実装する**。`/research`, `/plan`, `/implement`, `/review-assist`, `/status`のプロンプトを作成・テスト。
6. **ワークフローレベル基準を策定し、Standardレベルで試行する**。

### 評価と展開（Phase 2-3）

7. **Phase 1の定量・定性評価を実施し、拡大/継続/中止を判断する**。
8. **成功した場合、段階的に全チームへ展開する**。チームごとに1-2スプリントの移行期間を設ける。
9. **四半期レビューサイクルを確立する**。steering・コマンド・スキルの陳腐化チェックとBitter Lesson対策の簡素化レビューを定例化する。

## 付録B: SpecKit試験運用の教訓（本PJでの経験に基づく）

SpecKitの試験運用から得られた主要な教訓を、RPIフロー設計への反映事項として記録する。

| SpecKitでの問題 | 原因分析 | RPIフローでの対策 |
|---------------|---------|-----------------|
| 小規模修正にフル仕様プロセスが適用された | ワークフローレベルの分類機構がない | 3段階のワークフローレベル（Lightweight/Standard/Full） |
| 生成されるMarkdownが冗長 | テンプレートが網羅性を優先 | Plan文書のページ数制限（Standardは1ページ以内） |
| レビュアーの承認疲れ | 全フェーズにゲートが存在 | Lightweight/Standardではフェーズゲートを省略 |
| エージェントが仕様を無視 | コンテキスト過多 | 階層的情報管理＋タスク粒度の細分化 |
| コマンドの使い分けが分からない | 8コマンドの学習コスト | 5コマンドへの集約 |
